# Kestrel Relay Node (KRN-500) - Diagnostic Error Code Reference

## 1. Overview

This reference lists diagnostic codes emitted by the Kestrel Relay Node (KRN-500) firmware during normal sync operations. Each code below is self-contained: it names the triggering condition, the most common root cause, and the exact resolution behavior an operator or automated recovery script should expect. Codes are grouped by category and are unique across the full KRN-500 firmware line, so a given code always refers to the same condition regardless of which section it appears in below.

## 2. Sync and Timing Codes

Diagnostic code KR-4021 flags a heartbeat timeout between paired relay nodes; it is raised when a peer fails to acknowledge three consecutive keepalive frames sent at the standard interval. The condition typically follows a brief network partition or a peer restart. Once triggered, the relay places the peer into a quarantine state and stops routing new sync frames to it. Recovery is automatic: normal routing resumes after a fixed 45-second cooldown, provided a fresh heartbeat is received before the window closes. Fault marker KR-7734 appears when an inbound sync frame fails its checksum verification, which usually points to bit corruption introduced somewhere on the transport path rather than a problem with the sender itself. The relay discards the malformed frame immediately and requests retransmission from the origin node. If the origin does not deliver a clean retransmission within 4 attempts, the session is marked degraded and an operator alert is queued. Warning token KR-1198 is raised the moment a relay node's local clock drifts more than the tolerated skew from the cluster's reference time source. Clock drift beyond this bound can cause sync epochs to be ordered incorrectly, so the affected node is temporarily excluded from quorum voting until it resynchronizes. The tolerated skew is 200 milliseconds; once the node's offset falls back under that threshold for two consecutive checks, it rejoins quorum voting automatically without operator action.

## 3. Integrity and Verification Codes

Alert KR-5560 fires when the integrity ledger detects two conflicting write records for the same sync epoch, which almost always indicates a split-brain scenario following a network partition. The relay halts further writes to the affected epoch and escalates to manual review, since automatic conflict resolution is deliberately disabled for ledger conflicts. An operator must reconcile the two records using the ledger-diff tool and countersign the resolution with a change ticket before writes resume; unresolved KR-5560 alerts auto-escalate to a paging incident after 30 minutes. Condition KR-9082 marks a corrupted snapshot manifest discovered during a scheduled consistency sweep, meaning the manifest's recorded checksum no longer matches the recomputed value for one or more listed segments. The relay quarantines the snapshot and falls back to the previous known-good manifest automatically. Consistency sweeps run every 6 hours by default, so a fresh, valid snapshot is normally produced well before the quarantined one would have been needed for recovery. Signal KR-2317 indicates that a peer node's certificate fingerprint no longer matches the pinned value on file, which blocks the peer from completing the mutual TLS handshake required for sync traffic. This condition is most often caused by a certificate rotation that was not registered with the relay's trust store beforehand. To clear the fault, an operator must approve the new fingerprint through the trust-store console within 24 hours, after which the peer connection is retried automatically; fingerprints left unapproved past that window are purged and the peer must re-enroll from scratch.

## 4. Capacity and Throttling Codes

Notice KR-6649 is logged whenever outbound queue depth exceeds the configured watermark for longer than the grace period, signaling that a downstream peer is consuming sync frames more slowly than they are being produced. The default watermark is 5000 queued frames, and the grace period before KR-6649 is logged is 90 seconds of sustained overflow. Once logged, the relay begins shedding the lowest-priority frames first rather than blocking the entire pipeline. Throttle event KR-3305 occurs when a single client exceeds its allotted request budget within a rolling window, which protects the relay from any one client starving bandwidth from the rest of the fleet. The rolling window is 60 seconds wide, and the default per-client budget within that window is 400 requests. Clients that trip KR-3305 receive a backoff response and must wait out the remainder of the window before the budget resets. Capacity alarm KR-8814 triggers once the relay's local cache utilization crosses the high-water mark, which is set at 85 percent of the configured cache size by default. When the alarm is active, the relay begins evicting the least-recently-used entries until utilization falls back under 70 percent, the configured low-water mark, at which point normal caching resumes.

## 5. Access and Security Codes

Access fault KR-2266 is recorded when a connecting client presents an expired session token during the initial handshake, rather than a missing or malformed one, which is tracked separately under a different code. Session tokens issued by the relay are valid for 8 hours from issuance, and a client that fails with KR-2266 must request a fresh token from the auth endpoint before retrying the connection; retries with the same expired token are rejected outright rather than queued. Security event KR-5591 fires on three consecutive authentication failures from the same client identifier within a 10-minute window, and results in that identifier being locked out for 15 minutes before another attempt is permitted. The lockout is enforced per client identifier, not per source IP, so a client behind a shared address does not lock out its neighbors. Repeated lockouts within a 7-day period escalate to a security review. Audit flag KR-7743 appears when a configuration change is applied without a signed change ticket attached, which is permitted in development environments but blocked by policy in production. In production, two unresolved KR-7743 flags within a single calendar month automatically suspend further unsigned changes until an administrator reviews the audit log and manually clears the count. The audit log retains flagged entries for 180 days.

## 6. General Notes

All timers referenced above are measured from the moment the triggering condition is first detected, not from when the code is written to the log. Every code in this reference is emitted exactly once per occurrence; the relay does not repeat a code while the underlying condition remains active, though it may re-emit the same code if the condition clears and then recurs later. Support tickets referencing any code in this document should include the full code string, for example KR-4021, along with the firmware build number shown on the relay's status page.
